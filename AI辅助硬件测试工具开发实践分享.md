# AI 辅助硬件测试工具开发实践分享

## 一、项目背景

在显示设备（电视/显示器）的画质测试流程中，测试工程师需要反复执行以下操作：

1. 在信号发生器（Murideo Seven G8K）上设置测试图案（窗口大小、IRE 亮度、HDR/SDR 模式等）
2. 手动调整电视参数（图像模式、峰值亮度、背光值、Local Dimming 等）
3. 用色彩分析仪（Konica Minolta CA-410）读取亮度/色度值
4. 将数据手动记录到 Excel 表格

一个完整的峰值亮度测试往往涉及 **数百到数千个测试用例**，纯手动操作耗时长、易出错、难以复现。

本项目的目标：**用 AI（Claude Code）辅助开发一款自动化测试工具，将上述流程自动化，大幅提升测试效率。**

## 二、AI 辅助开发的核心能力

在整个开发过程中，Claude Code 展现了以下关键能力：

### 2.1 逆向工程未知硬件协议

Murideo Seven G8K 的串口/WebSocket 协议**没有公开文档**，官方只提供了一个基础协议说明，关键的 IRE 窗口控制命令完全未知。AI 辅助完成了以下逆向工作：

| 阶段 | 方法 | AI 的角色 |
|------|------|-----------|
| 协议发现 | 编写 WebSocket 嗅探器（`sniff_murideo_ws.py`），捕获 Murideo Web UI 发出的命令 | AI 根据需求编写嗅探脚本，解析捕获的二进制数据 |
| 命令格式验证 | 编写多个测试脚本（`test_ire_window.py` / `test_ire_window2.py` / `test_ire_window3.py`） | AI 快速迭代不同参数组合和时序方案 |
| 时序确定 | 测试命令间的延迟要求 | AI 发现 700ms 延迟是 Window 图案命令到 IRE 命令的必要间隔 |
| 参数范围 | 确认 IRE 值范围（0-255）和窗口大小范围（0-100%） | AI 设计系统性测试矩阵，逐项验证 |

最终发现的命令序列：

```
1. SENDOTHER||63739              → 初始化 IRE 模式
2. SENDDOUBLE||98,26             → 选择 Window 图案
3. 等待 700ms
4. SENDOTHER||30971,{IRE},{SIZE} → 设置 IRE 亮度 + 窗口大小
```

### 2.2 串口通信驱动开发

CA-410 色彩分析仪使用 CA-S40 串口协议，有一个特殊的数据格式——色度值 x/y **省略 "0." 前缀**（如 "230" 表示 0.230）。AI 根据实际设备响应：

```
OK00,P1 230;182;12.4    → x=0.230, y=0.182, Lv=12.4
```

自动识别了这一格式并实现了正确的解析逻辑。

## 三、关键技术实现

### 3.1 双通道传输层设计

Murideo 支持 WebSocket 和串口两种控制方式，我们实现了统一的传输抽象：

**WebSocket 传输**：
- 持久化连接 + 后台 asyncio 事件循环
- 30 秒心跳保活
- 断线自动重连（指数退避，最多 3 次）
- 批量命令发送（关键：IRE 模式状态不跨连接保持）

**串口传输**：
- 回声过滤（设备会回显发送的命令）
- 分帧读取（处理数据碎片和超时）
- 多行响应解析（返回最后一个 `RESPONSE||` 行）

```python
# 统一的传输接口
class MurideoTransport(ABC):
    def connect(self, **kwargs) -> None: ...
    def disconnect(self) -> None: ...
    def send_and_recv(self, command, timeout) -> str | None: ...
    def send_batch(self, commands, delays, timeout) -> list[str | None]: ...
```

### 3.2 生产者-消费者线程模型

所有硬件 I/O 操作在后台守护线程执行，通过 `queue.Queue` 与 UI 线程通信：

```
用户点击 → Worker线程执行驱动调用 → 结果推入Queue → UI轮询(100ms) → 界面更新
```

关键设计：
- `MeasurementWorker`：管理 CA-410 连接/测量/校准/连续测量
- `AutoTestWorker`：编排完整测试流程（设置 Murideo → 等待稳定 → 触发测量 → 写入 Excel → 下一步）
- TV 参数变更检测：当图像模式/峰值亮度/背光等变化时，自动弹出确认对话框

### 3.3 自动化测试流程

```
AutoTestWorker 每个测试用例的执行流程：

1. 检测 TV 参数是否变化 ──→ 是 ──→ 弹窗等待用户确认
2. 设置 Murideo 参数（仅发送变化的参数，减少命令数）
3. 等待显示稳定（可配置，默认 3 秒）
4. 触发 CA-410 测量
5. 等待测量结果返回
6. 自动写入 Excel（匹配测试用例行，填入 Lv/x/y 值）
7. 等待步骤间隔（可配置，默认 2 秒）
8. 进入下一个用例
```

智能优化：
- **增量命令发送**：仅发送与上一步不同的参数，显著减少通信量
- **TV 参数变更检测**：只在图像模式/峰值亮度/背光/LD 变化时才打断用户
- **暂停/恢复/停止**：全程可控，支持中断后继续

### 3.4 Excel 双向集成

**读取**：从 Excel "测试用例" 工作表加载测试步骤（图像模式、窗口大小、IRE 值等）

**写入**：每次测量后，按 7 个关键字段匹配 Excel 中的行，将 Lv/x/y 写入对应位置；未匹配的数据追加到 "额外数据" 工作表

```python
# 匹配逻辑：7 个字段完全一致才算命中
匹配字段: image_mode, peak_brightness, backlight_value,
          local_dimming, window_size, HDR/SDR, window_brightness
写入字段: Lv (cd/m²), x, y
```

### 3.5 测试方案系统

JSON 格式的测试方案定义了 Murideo 默认参数和测试步骤：

```json
{
  "name": "HDR Peak Brightness Test",
  "defaults": { "hdr": 1, "timing": 34, "color_space": 0, "color_depth": 1 },
  "steps": [
    { "window_size": 10, "ire": 255, "note": "HDR10 10%窗口 100%IRE" },
    { "window_size": 50, "ire": 255, "note": "HDR10 50%窗口 100%IRE" },
    { "hdr": 0, "window_size": 100, "ire": 255, "note": "SDR 100%窗口 100%IRE" }
  ]
}
```

步骤中未指定的字段自动继承 `defaults`，减少冗余。

## 四、AI 辅助开发的效率提升

### 4.1 定量对比

| 指标 | 传统开发 | AI 辅助开发 |
|------|---------|------------|
| 协议逆向 | 数天手动抓包+试错 | 2-3 轮迭代脚本，半天完成 |
| 驱动开发 | 查阅文档+逐步调试 | 一次生成框架，微调特殊格式 |
| UI 开发 | 手写布局代码 | 描述需求，AI 生成完整界面 |
| 测试用例生成 | 手写 Excel 公式 | AI 生成 Python 脚本，秒级产出 |
| 整体项目 | 2-3 周 | **3 天** |

### 4.2 AI 特别擅长的场景

1. **格式解析**：面对 CA-410 省略 "0." 前缀的特殊格式，AI 从原始数据中快速归纳出规律
2. **协议逆向**：编写嗅探器 → 分析流量 → 设计验证实验 → 确认命令格式，整个过程 AI 高效串联
3. **跨域整合**：串口通信 + WebSocket + tkinter UI + Excel 读写 + 多线程，涉及多个技术栈的融合
4. **边界条件处理**：Murideo 的回声过滤、分帧读取、IRE 模式状态不跨连接等细节，AI 都能考虑到

### 4.3 人机协作模式

```
人类负责：                    AI 负责：
- 定义需求和测试目标           - 编写实现代码
- 确认协议逆向结果             - 生成测试和验证脚本
- 手动验证关键功能             - 处理边界条件和异常
- 做架构决策                   - 快速迭代和试错
                              - 生成文档和用例
```

## 六、经验总结

### 6.1 AI 辅助开发的最佳实践

1. **先让 AI 写探索脚本**：面对未知硬件，让 AI 快速生成探测/嗅探代码，比手动试错效率高一个数量级
2. **提供真实数据给 AI**：把设备实际返回的原始数据（如 `OK00,P1 230;182;12.4`）粘贴给 AI，让它归纳格式规律
3. **分模块迭代**：先驱动层 → 再 Worker 层 → 最后 UI 层，每层验证后再推进
4. **利用 AI 生成测试矩阵**：多因素交叉测试用例用 Python 脚本生成，比手写 Excel 公式更可靠
5. **保留调试工具**：`tools/` 目录下的探测脚本本身就是 AI 辅助逆向的产物，也是后续排障的利器

### 6.2 注意事项

1. **AI 生成的协议逻辑必须实测验证**：AI 推断的命令格式可能不完全正确，需要用真实设备确认
2. **时序问题需要特别注意**：硬件通信的延迟要求（如 700ms）是 AI 容易忽略的，需要人工反馈和调整
3. **错误处理需要实测补充**：AI 生成的错误处理覆盖常见情况，但硬件特有的异常需要实测中发现和补充
4. **跨平台兼容性**：CJK 字体、串口名称等平台差异需要在不同环境测试
